# Repository Instructions
This repository documents and supports the ongoing development of a cybersecurity portfolio project centered on a SOC analyst flagship lab.

This repository documents a live home lab using:

- Splunk Enterprise
- Splunk Universal Forwarder
- Wazuh Manager and Agent
- Sysmon
- Windows Security Event Logs
- Windows 11 monitored endpoint
- Ubuntu centralized monitoring server
- Kali Linux controlled attacker system
- Chrony/NTP time synchronization

Important project rules:

1. Do not fabricate evidence, attack results, screenshots, event output, validation results, false positives, or completed investigations.
2. Do not claim a detection is production-ready.
3. Clearly distinguish:
   - Planned
   - Draft
   - Experimental
   - Validated
   - Completed
4. Preserve my existing technical content unless it is clearly duplicated, empty, invalid, or moved to a more appropriate file.
5. Use `git mv` when moving tracked files so Git preserves their history.
6. Do not delete non-empty files without explaining why.
7. Before modifying anything, inspect the repository and provide:
   - Current file tree
   - Proposed file tree
   - Files to move
   - Files to rename
   - Files that appear empty
   - Files that appear invalid
   - Files that should not be changed yet
8. Work in phases. Complete only the phase I request.
9. After each phase:
   - Show every file changed
   - Summarize what changed
   - Check relative Markdown links
   - Check `git status`
   - Identify anything that still requires manual verification
10. Do not insert secrets, passwords, API keys, tokens, public IP addresses, or personally identifying information.
11. Private RFC1918 lab IP addresses may remain where technically useful, but point out where placeholders might be more appropriate.
12. Do not convert planned capabilities into completed claims.
13. Keep the writing professional but grounded in what this lab actually contains.
14. When describing an enterprise-equivalent workflow, distinguish it from what was actually performed in the home lab.
15. Favor accurate, readable documentation over excessive corporate language.
16. Do not generate generic filler files simply to make the repository appear larger.
17. Use GitHub-compatible relative links and verify that their capitalization matches the actual filenames.
18. Use lowercase kebab-case for new folders and filenames unless a conventional filename such as README.md, LICENSE, or CHANGELOG.md requires otherwise.
